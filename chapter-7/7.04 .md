## 7.4 Interrupt-Driven I/O

ปัญหาของ Programmed I/O คือโปรเซสเซอร์ต้องรอเป็นเวลานานสำหรับโมดูล I/O ที่เกี่ยวข้องให้พร้อมสำหรับการรับหรือการส่งข้อมูล ขณะรอ โปรเซสเซอร์จะต้องสอบถามสถานะของโมดูล I/O ซ้ำ ๆ ส่งผลให้ประสิทธิภาพของระบบทั้งหมดลดลงอย่างมาก

ทางเลือกอื่นคือให้โปรเซสเซอร์ออกคำสั่ง I/O ไปยังโมดูลแล้วดำเนินงานที่มีประโยชน์อื่นต่อไป โมดูล I/O จะขัดจังหวะโปรเซสเซอร์เพื่อร้องขอบริการเมื่อพร้อมที่จะแลกเปลี่ยนข้อมูลกับโปรเซสเซอร์ จากนั้นโปรเซสเซอร์จะดำเนินการถ่ายโอนข้อมูลและกลับมาดำเนินการที่ค้างอยู่

### การทำงานจากมุมมองของโมดูล I/O

สำหรับการรับเข้า โมดูล I/O ได้รับคำสั่ง READ จากโปรเซสเซอร์ จากนั้นโมดูล I/O ดำเนินการอ่านข้อมูลจากอุปกรณ์ต่อพ่วงที่เกี่ยวข้อง เมื่อข้อมูลอยู่ในรีจิสเตอร์ข้อมูลของโมดูล โมดูลจะส่งสัญญาณอินเทอร์รัปต์ไปยังโปรเซสเซอร์ผ่านสายควบคุม โมดูลจะรอจนกว่าโปรเซสเซอร์จะร้องขอข้อมูล เมื่อมีคำขอ โมดูลจะวางข้อมูลบนบัสข้อมูลและพร้อมสำหรับการดำเนินการ I/O อื่น

### การทำงานจากมุมมองของโปรเซสเซอร์

โปรเซสเซอร์ออกคำสั่ง READ แล้วดำเนินงานอื่น (เช่น โปรเซสเซอร์อาจทำงานกับหลายโปรแกรมในเวลาเดียวกัน) ในตอนท้ายของแต่ละรอบคำสั่ง โปรเซสเซอร์จะตรวจสอบอินเทอร์รัปต์ เมื่ออินเทอร์รัปต์จากโมดูล I/O เกิดขึ้น โปรเซสเซอร์จะบันทึกบริบท (เช่น ตัวนับโปรแกรมและรีจิสเตอร์โปรเซสเซอร์) ของโปรแกรมปัจจุบันและประมวลผลอินเทอร์รัปต์ ในกรณีนี้ โปรเซสเซอร์อ่านคำข้อมูลจากโมดูล I/O และเก็บไว้ในหน่วยความจำ จากนั้นคืนค่าบริบทของโปรแกรมที่กำลังทำงานอยู่และกลับมาดำเนินการต่อ

Interrupt I/O มีประสิทธิภาพมากกว่า Programmed I/O เนื่องจากช่วยลดการรอที่ไม่จำเป็น อย่างไรก็ตาม Interrupt I/O ยังคงใช้เวลาโปรเซสเซอร์จำนวนมาก เนื่องจากข้อมูลทุกคำที่ส่งระหว่างหน่วยความจำกับโมดูล I/O จะต้องผ่านโปรเซสเซอร์

### การประมวลผลอินเทอร์รัปต์

เมื่อการดำเนินการ I/O ของอุปกรณ์เสร็จสมบูรณ์ จะเกิดลำดับเหตุการณ์ฮาร์ดแวร์ต่อไปนี้:

1. อุปกรณ์ส่งสัญญาณอินเทอร์รัปต์ไปยังโปรเซสเซอร์
2. โปรเซสเซอร์เสร็จสิ้นการดำเนินการคำสั่งปัจจุบันก่อนที่จะตอบสนองต่ออินเทอร์รัปต์
3. โปรเซสเซอร์ทดสอบอินเทอร์รัปต์ พิจารณาว่ามีอินเทอร์รัปต์ และส่งสัญญาณยืนยัน (Acknowledgment) ไปยังอุปกรณ์ที่ออกอินเทอร์รัปต์ การยืนยันอนุญาตให้อุปกรณ์ลบสัญญาณอินเทอร์รัปต์ของตนได้
4. โปรเซสเซอร์จำเป็นต้องเตรียมการถ่ายโอนการควบคุมไปยังรูทีนอินเทอร์รัปต์ โดยจะต้องบันทึกข้อมูลขั้นต่ำที่จำเป็น ได้แก่: (ก) สถานะของโปรเซสเซอร์ที่อยู่ในรีจิสเตอร์ที่เรียกว่า Program Status Word (PSW) และ (ข) ตำแหน่งของคำสั่งถัดไปที่จะดำเนินการซึ่งอยู่ในตัวนับโปรแกรม ข้อมูลเหล่านี้จะถูกผลักเข้าสู่สแต็กควบคุมระบบ
5. โปรเซสเซอร์โหลดตัวนับโปรแกรมด้วยตำแหน่งเริ่มต้นของโปรแกรมจัดการอินเทอร์รัปต์ที่จะตอบสนองต่ออินเทอร์รัปต์นี้

เมื่อตัวนับโปรแกรมถูกโหลดแล้ว โปรเซสเซอร์จะดำเนินไปสู่รอบคำสั่งถัดไปและการควบคุมจะถูกถ่ายโอนไปยังโปรแกรมจัดการอินเทอร์รัปต์ การดำเนินการของโปรแกรมนี้ส่งผลให้เกิดการดำเนินการต่อไปนี้:

6. โปรแกรมจัดการอินเทอร์รัปต์จะเริ่มต้นด้วยการบันทึกเนื้อหาของรีจิสเตอร์ทั้งหมดบนสแต็ก
7. โปรแกรมจัดการอินเทอร์รัปต์ประมวลผลอินเทอร์รัปต์ ซึ่งรวมถึงการตรวจสอบข้อมูลสถานะที่เกี่ยวข้องกับการดำเนินการ I/O และอาจส่งคำสั่งหรือการยืนยันเพิ่มเติมไปยังอุปกรณ์ I/O
8. เมื่อการประมวลผลอินเทอร์รัปต์เสร็จสมบูรณ์ ค่ารีจิสเตอร์ที่บันทึกไว้จะถูกดึงจากสแต็กและคืนค่าไปยังรีจิสเตอร์
9. การกระทำสุดท้ายคือการคืนค่า PSW และตัวนับโปรแกรมจากสแต็ก ส่งผลให้คำสั่งถัดไปที่จะดำเนินการมาจากโปรแกรมที่ถูกขัดจังหวะก่อนหน้า

### ประเด็นการออกแบบ

ประเด็นการออกแบบสองประเด็นเกิดขึ้นในการใช้งาน Interrupt I/O:
1. เมื่ออินเทอร์รัปต์เกิดขึ้น โปรเซสเซอร์จะพิจารณาได้อย่างไรว่าอุปกรณ์ใดออกอินเทอร์รัปต์
2. หากเกิดอินเทอร์รัปต์หลายรายการ โปรเซสเซอร์จะตัดสินใจประมวลผลรายการใดก่อน

มีเทคนิคสี่ประเภททั่วไปที่ใช้:

**1. สายอินเทอร์รัปต์หลายเส้น (Multiple Interrupt Lines):** วิธีตรงที่สุดคือการมีสายอินเทอร์รัปต์หลายเส้นระหว่างโปรเซสเซอร์กับโมดูล I/O อย่างไรก็ตาม ไม่สามารถทุ่มเทสายบัสหรือขาโปรเซสเซอร์มากกว่าสองสามเส้นให้กับสายอินเทอร์รัปต์ได้

**2. การสำรวจซอฟต์แวร์ (Software Poll):** เมื่อโปรเซสเซอร์ตรวจจับอินเทอร์รัปต์ จะแยกไปที่รูทีนบริการอินเทอร์รัปต์ที่สำรวจแต่ละโมดูล I/O เพื่อพิจารณาว่าโมดูลใดเป็นสาเหตุของอินเทอร์รัปต์

**3. Daisy Chain (การสำรวจฮาร์ดแวร์ แบบเวกเตอร์):** โมดูล I/O ทั้งหมดใช้สายร้องขออินเทอร์รัปต์ร่วมกัน สายยืนยันอินเทอร์รัปต์เชื่อมต่อแบบ Daisy Chain ผ่านโมดูล เมื่อโปรเซสเซอร์ตรวจจับอินเทอร์รัปต์ จะส่งสัญญาณยืนยันอินเทอร์รัปต์ออกไป สัญญาณนี้จะแพร่ผ่านโมดูล I/O จนกว่าจะถึงโมดูลที่ร้องขอ โมดูลที่ร้องขอโดยทั่วไปจะตอบสนองโดยวางคำบนสายข้อมูล คำนี้เรียกว่าเวกเตอร์ (Vector) และเป็นที่อยู่ของโมดูล I/O หรือตัวระบุเฉพาะอื่น ๆ เทคนิคนี้เรียกว่า **Vectored Interrupt**

**4. การอนุญาโตตุลาการบัส (Bus Arbitration แบบเวกเตอร์):** โมดูล I/O ต้องได้รับการควบคุมบัสก่อนจึงจะสามารถยกสายร้องขออินเทอร์รัปต์ได้ ดังนั้น มีเพียงโมดูลเดียวเท่านั้นที่สามารถยกสายนี้ได้ในแต่ละครั้ง

### ตัวควบคุมอินเทอร์รัปต์ Intel 82C59A

Intel 80386 มีสาย Interrupt Request (INTR) และ Interrupt Acknowledge (INTA) เพียงเส้นเดียว เพื่อให้ 80386 จัดการกับอุปกรณ์ที่หลากหลายและโครงสร้างลำดับความสำคัญ จึงมักกำหนดค่าด้วยตัวกลางอินเทอร์รัปต์ภายนอก 82C59A อุปกรณ์ภายนอกเชื่อมต่อกับ 82C59A ซึ่งเชื่อมต่อกับ 80386 ต่อไป

82C59A ตัวเดียวสามารถจัดการได้สูงสุดแปดโมดูล หากต้องการควบคุมมากกว่าแปดโมดูล สามารถใช้การจัดเรียงแบบ Cascade เพื่อจัดการได้ถึง 64 โมดูล

82C59A มีหน้าที่รับผิดชอบด้านการจัดการอินเทอร์รัปต์เพียงอย่างเดียว โดยรับคำขออินเทอร์รัปต์จากโมดูลที่เชื่อมต่อ พิจารณาว่าอินเทอร์รัปต์ใดมีลำดับความสำคัญสูงสุด จากนั้นส่งสัญญาณโปรเซสเซอร์โดยยกสาย INTR

82C59A สามารถโปรแกรมได้ มีโหมดอินเทอร์รัปต์ดังต่อไปนี้:

- **Fully Nested:** คำขออินเทอร์รัปต์ถูกเรียงตามลำดับความสำคัญจาก IR0 ถึง IR7
- **Rotating:** ในโหมดนี้ อุปกรณ์ที่ได้รับบริการแล้วจะได้รับลำดับความสำคัญต่ำสุดในกลุ่ม
- **Special Mask:** ให้โปรเซสเซอร์ยับยั้งอินเทอร์รัปต์จากอุปกรณ์บางชนิด

### อินเทอร์เฟซต่อพ่วงโปรแกรมได้ Intel 8255A

Intel 8255A Programmable Peripheral Interface เป็นโมดูล I/O อเนกประสงค์บนชิปเดียว ออกแบบมาเพื่อใช้กับโปรเซสเซอร์ Intel 80386 โดยมีสายสัญญาณดังต่อไปนี้:

- **D0–D7:** สายรับเข้าและส่งออกข้อมูล ข้อมูลทั้งหมดที่อ่านจากและเขียนไปยัง 8255A ผ่านสายข้อมูลแปดเส้นนี้
- **CS (Chip Select Input):** หากสายนี้มีค่าลอจิก 0 ไมโครโปรเซสเซอร์สามารถอ่านและเขียนไปยัง 8255A ได้
- **RD (Read Input):** หากสายนี้มีค่าลอจิก 0 และอินพุต CS มีค่าลอจิก 0 เอาต์พุตข้อมูล 8255A จะถูกเปิดใช้งานบนบัสข้อมูลระบบ
- **WR (Write Input):** หากสายอินพุตนี้มีค่าลอจิก 0 และอินพุต CS มีค่าลอจิก 0 ข้อมูลจะถูกเขียนไปยัง 8255A จากบัสข้อมูลระบบ
- **RESET:** 8255A จะอยู่ในสถานะรีเซ็ตหากสายอินพุตนี้มีค่าลอจิก 1 พอร์ตต่อพ่วงทั้งหมดจะถูกตั้งค่าเป็นโหมดอินพุต
- **PA0–PA7, PB0–PB7, PC0–PC7:** สายสัญญาณเหล่านี้ใช้เป็นพอร์ต I/O 8 บิต สามารถเชื่อมต่อกับอุปกรณ์ต่อพ่วงได้
- **A0, A1:** การรวมกันทางลอจิกของสายอินพุตสองเส้นนี้กำหนดว่ารีจิสเตอร์ภายในใดของ 8255A ที่ข้อมูลจะถูกเขียนไปหรืออ่านจาก

สาย I/O 24 เส้นถูกแบ่งออกเป็นสามกลุ่ม 8 บิต (A, B, C) แต่ละกลุ่มสามารถทำหน้าที่เป็นพอร์ต I/O 8 บิต ให้การเชื่อมต่อสำหรับอุปกรณ์ต่อพ่วงสามชนิด นอกจากนี้ กลุ่ม C ยังถูกแบ่งย่อยออกเป็นกลุ่ม 4 บิต (CA และ CB)

โปรเซสเซอร์ควบคุม 8255A โดยใช้รีจิสเตอร์ควบคุม 8 บิต สามารถตั้งค่าได้สามโหมด:

- **Mode 0:** โหมด I/O พื้นฐาน สามกลุ่มของสาย 8 เส้นภายนอกทำหน้าที่เป็นพอร์ต I/O 8 บิตสามพอร์ต แต่ละพอร์ตสามารถกำหนดเป็นอินพุตหรือเอาต์พุต
- **Mode 1:** พอร์ต A และ B สามารถกำหนดเป็นอินพุตหรือเอาต์พุต และสายจากพอร์ต C ทำหน้าที่เป็นสายควบคุมสำหรับ A และ B สัญญาณควบคุมทำหน้าที่สองประการหลักคือ "Handshaking" และการร้องขออินเทอร์รัปต์
- **Mode 2:** โหมดสองทิศทาง พอร์ต A สามารถกำหนดเป็นสายอินพุตหรือเอาต์พุตสำหรับการรับส่งข้อมูลสองทิศทางบนพอร์ต B โดยสาย B ให้ทิศทางตรงข้าม
